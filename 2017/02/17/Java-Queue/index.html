Queue接口

发表于 2017-02-17   |   分类于 Java ， JDK   |     |   阅读次数

一个为存储即将处理的元素的集合类，除了基本的Collection操作，队列(queue)还提供插入、取出、检查(读)操作，每个方法都有两种形式，程序出错时抛出异常或者返回一个特殊值(null或是false)，后者的插入操作专为有容量限制的Queue类使用，在绝大多数实现中，插入不会失败。

Queue<E>接口

所属包 java.util
扩展了Collection<E> ->Iterable<E>
子接口: BlockingDeque<E>, BlockingQueue<E>, Deque<E>, TransferQueue<E>
实现类: AbstractQueue, ArrayBlockingQueue, ArrayDeque, ConcurrentLinkedDeque,
ConcurrentLinkedQueue, DelayQueue, LinkedBlockingDeque, LinkedBlockingQueue, LinkedList,
LinkedTransferQueue, PriorityBlockingQueue, PriorityQueue, SynchronousQueue
Java 1.5 开始使用

以下是该接口方法的小结:

Throws exception Returns special value
Insert add(e) offer(e)
Remove remove() poll()
Examine element() peek()

          队列通常(但非一定)以先入先出(FIFO)的方式提供元素，抛出异常的队列对象包括优先队列和栈。优先队列根据某个提供的比较器或者是元素的自然顺序提供元素。栈以后入先出(LIFO)的方式运转。但无论何种方式，队列的首部元素通过调用remove()或者poll()删除，在FIFO队列中，所有的新元素被插入队列尾，其他形式的队列可能使用不同的放置规则。每个Queue类的实现都需要指出它处理元素的规则。
          offer方法如果插入元素失败，则返回false，这与Collection.add方法不同，后者只会抛出一个unchecked异常，offer方法用在对待元素插入失败很正常的情况，譬如空间限制的队列。
remove()和poll()方法移除并返回队首元素，但还是要看具体的实现策略。只有在队列为空时两者有区别：remove()抛出异常，而poll()返回null。
          element()和peek()方法返回(不移除)队首元素。
          Queue接口没有定义blocking queue方法(并发编程中常见)，这些方法等待元素插入队列或者元素被取走。相关方法在BlockingQueue接口中定义，该接口扩展了当前接口。
          Queue接口实现通常不允许插入空元素，尽管一些类的实现(譬如LinkedList)并没有限制。但这样的禁止的有用的，否则无法区分poll()返回的null是何缘由。
          Queue接口的实现一般不规定以元素为基础的equals方法与hashCode方法，而是继承类对象本身规定的equals方法与hashCode方法。因为以元素为基础的相等比较在遇到”相同元素但不同顺序的队列”的情况时效果不佳。

方法

对父类方法有扩充含义的方法:

  • boolean add(E e): 向队列插入元素，成功则返回true。如果有容量限制且没有空间空余，则抛出IllegalStateException。

以下列出了除去父类Collection<E>的方法:

  • boolean offer(E e): 向队列插入元素，如果队列空间受限，则返回false。该方法优于add(E)方法(抛出异常)。(但这并不意味该方法不抛出异常!)
  • E remove(): 返回并移除队列首元素。该方法与poll()方法不同之处在于，如果队列为空，remove()将抛出NoSuchElementException。(该方法与Collection接口的remove()方法不同，它不带任何参数)
  • E poll(): 返回并移除队列首元素。如果队列为空，则返回null。
  • E element(): 返回但不移除队首元素。该方法与peek()方法不同之处在于，如果队列为空，element()将抛出NoSuchElementException。
  • E peek(): 返回但不移除队首元素，如果队列为空则返回null。
