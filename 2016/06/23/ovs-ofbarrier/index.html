Open vSwitch 处理OpenFlow的Barrier消息

发表于 2016-06-23   |   分类于 laboratory ， OpenvSwitch   |     |   阅读次数

Barrier消息从OpenFlow0.9版本开始启用，用以保证消息的有序执行。控制器使用 Barrier request/reply消息，来确保之后执行的消息所依赖的已经全部执行，或是为了收到操作完成的通知。前者也就是说，假设有两条来自控制器的消息: 添加流表消息，和依靠该流表下发的数据包，就可以在其中插入Barrier消息，用以确认数据包下发在流表下发之后执行。

当交换机收到一条Barrier消息后，首先它必须完成收到的所有的命令，才能执行Barrier消息以后的命令，当Barrier消息之前的所有命令全部执行了以后，交换机必须向控制器回复相应的Barrier Reply消息(xid相同)。Barrier消息没有其他内容。

本博客的研究环境: Open vSwitch 2.5.0、OpenFlow 1.3
部分OpenFlow讲解参考之前的文章。

OpenFlow中的Barrier消息

OpenFlow中关于Barrier的介绍：When the controller wants to ensure message dependencies have been met or wants to receive notifications for completed operations, it may use an OFPT_BARRIER_REQUEST message. This message has no body. Upon receipt, the switch must finish processing all previously-received messages, including sending corresponding reply or error messages, before executing any messages beyond the Barrier Request. When such processing is complete, the switch must send an OFPT_BARRIER_REPLY message with the xid of the original request.

出现引用的文件: vconn.c、ofctrl.c、ovs-ofctl.c、ofp-util.c。 ofctl中做的工作也许和aux的barrier有关。

  • ofp-util.c 有ofputil_encode_barrier_request()函数。
  • vconn.c 涉及barrier: 向vconn发送请求，随后发送一个barrier消息，然后vconn堵塞，直到收到一个barrier回复，用于发送不需要回复的OpenFlow请求时，判断在特殊情况下是否请求出错。
基本工作流程

交换机收到消息后，分析消息类型，如果是OFPTYPE_BARRIER_REPLY则不予处理，如果是OFPTYPE_BARRIER_REQUEST则执行函数handle_barrier_request(ofconn, oh)

ofpraw_alloc_reply()函数生成了对应于request消息的reply。与ofpraw_alloc()函数类似，但包括OpenFlow版本(raw代表版本号，来源于request->version)和传输ID(根据request->xid获得)
