OpenFlow 1.3 的一些基础知识

发表于 2016-06-21   |   分类于 laboratory ， OpenvSwitch   |     |   阅读次数

OpenFlow 信道

OpenFlow支持三类消息: 控制器-交换机消息、异步消息、对称消息。

  • 控制器-交换机消息由控制器初始化，用于直接管理或监测交换机的状态。
  • 异步消息由交换机初始化，用于更新控制器中的网络事件和交换机状态的改变。
  • 对称消息既可以被交换机初始化，也可以被控制器初始化，不需要询问信息。
控制器-交换机信息

控制器-交换机信息由控制器发送，可以需要交换机的回复，也可以不需要。包括Features, Configuration, Modify-State, Read-State, Packet-out, Barrier, Role-Request, Asynchronous-Configuration。

异步消息

异步消息的发送不需要控制器的请求，交换机向控制器发送该消息以说明数据包的到达、交换机状态的改变或是出错。包括Packet-in, Flow-Removed, Port-status, Error。

对称消息

对称消息可由连接的任一方发出，不需要请求。包括Hello, Echo, Experimenter。

OpenFlow 连接建立流程

OpenFlow协议提供可靠的消息传递和处理，但并不提供消息确认确认或保证消息的有序处理。OpenFlow在主要连接和辅助连接上的消息处理是采用了可靠传输，辅助连接不支持不可靠的传输。有序处理可以通过Barrier消息保证。
OpenFlow建立在TCP/IP连接之上。OpenFlow的连接主要注意以下几个方面:

连接设置

交换机必须能够与一个用户可以配置IP地址(其他属性固定)的控制器建立连接，端口号也由用户指定。交换机知道控制器的IP地址以后即可发起与控制器的TLS或TCP连接。
OpenFlow连接一旦建立，连接双方都要立刻发送OFPT_HELLO消息，其中包含发送者支持的OpenFlow协议最高版本号，接收者比较该版本号是否高于自己支持的版本号，如果不高于，则连接继续，否则接收者需要回复OFPT_ERROR消息，type字段为OFPET_HELLO_FAILED，code字段为OFPHFC_COMPATIBLE，选择性地用ASCII字符串填写data字段，用以解释当前状态，然后终止连接。

连接中断

在交换机失去与所有控制器的连接(echo请求超时、TLS会话超时或者其他端口连接)后，交换机应立刻进入”fail secure mode”或者”standalone mode” (进入两种模式的哪一种依赖交换机的实现和配置)。
在”fail secure mode”模式下，交换机行为的唯一变化就是所有到控制器的数据包或消息全部丢弃。流表项(Flow entries)根据他们的实效继续执行知道期满。
在”fail standalone”模式下，交换机使用OFPP_NORMAL保留端口处理所有的包，也就是说交换机此时充当了传统以太网中的交换机或路由器，这种模式通常只适用于Hybrid交换机。
在重新连接控制器后，流表项全部保留，控制器根据需求来选择是否丢弃所有的流表项。
交换机第一次启动后会进入”fail secure mode”或者”fail standalone”模式，直到与控制器成功地建立了连接。

加密

交换机与控制器可能通过TLS连接进行对话。TLS由交换机初始化，在控制器上启动，默认在TCP 6633端口。交换机和控制器通过交换由私钥签署的证书进行验证，每个交换机都要用户配置一个证书用来验证控制器(控制器证书)，另一个用于控制器的认证(交换机证书)。
交换机与控制器的连接也可以通过TCP连接，建议使用其他的安全措施，以防止监听、控制器假冒或者其他的OpenFlow信道攻击。

多个控制器

交换机连接多个控制器，主要是解决控制器失效或者负载均衡。
交换机同时连接所有设置的控制器。
